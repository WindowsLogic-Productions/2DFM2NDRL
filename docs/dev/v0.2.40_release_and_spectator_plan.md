# v0.2.40 release + spectator iteration plan

Snapshot from the 2026-05-13 working session. Captures (a) what's
unreleased on `main` since v0.2.39, (b) how to verify the new
auto-upload pipeline before shipping, (c) the runbook for testing
remote spectate, and (d) the three-iteration plan for making spectator
"mid-CSS" and "mid-battle" joins feel right.

---

## 2. Verifying the auto-upload pipeline before shipping

Before tagging v0.2.40, walk through this end-to-end. If any step
fails, fix it; don't ship a half-broken telemetry pipe.

### Server-side smoke (already passed)

This was verified during the build session. Confirm it's still up
post-cron-run / post-reboot:

```bash
curl https://hub.2dfm.org/logs/healthz
# Expect:  ok
```

If that times out: `ssh 2dfm-root 'systemctl status fm2k-logs caddy'`.

### End-to-end with a fake upload

The `tools/fm2k_logs.py` pulldown tool reads the secret from
`~/.config/fm2k/log_secret`. Manual end-to-end probe that doesn't
require a real desync:

```bash
SECRET=$(cat ~/.config/fm2k/log_secret)
cd /tmp && mkdir -p probe && cd probe
echo "synthetic test log" > FM2K_P1_Debug.log
echo "synthetic desync diff" > FM2K_P1_desync_f1.log

curl -s -H "X-FM2K-Log-Secret: $SECRET" \
  -F 'meta={"kind":"desync","session_id":"1234567890","match_id":"manual-probe","player_index":0,"client_version":"manual-probe","game_id":"WonderfulWorld_ver_0946","frame":1,"timestamp":"2026-05-13T00:00:00Z"}' \
  -F "files=@FM2K_P1_Debug.log" \
  -F "files=@FM2K_P1_desync_f1.log" \
  https://hub.2dfm.org/logs/upload

# Then verify via the dev tool:
/mnt/c/dev/wanwan/tools/fm2k_logs.py recent --limit 5
/mnt/c/dev/wanwan/tools/fm2k_logs.py show 1234567890

# Clean up:
ssh 2dfm-root 'rm -rf /var/lib/fm2k_logs/$(date -u +%Y-%m-%d)/1234567890 && \
  /home/fm2k/logs/venv/bin/python3 -c "import sqlite3; db=sqlite3.connect(\"/var/lib/fm2k_logs/index.sqlite\"); db.execute(\"DELETE FROM uploads WHERE session_id=?\", (\"1234567890\",)); db.commit()"'
```

If `recent` doesn't show your probe: the server stored it but the
index wasn't written. Check `journalctl -u fm2k-logs -n 50` on the VPS
for an exception in `_open_db()`.

### End-to-end with a real built launcher

This is the part that catches client-side bugs:

1. Build with the secret baked in:
   ```bash
   export FM2K_LOG_UPLOAD_SECRET="$(cat ~/.config/fm2k/log_secret)"
   ./go.sh
   ```

2. Run the launcher. Open the developer section. Confirm:
   - **"Auto-upload crash/desync diagnostics"** checkbox is visible and
     **clickable** (not greyed out). If greyed: `auto_upload_secret.h`
     was generated with an empty string — env var wasn't set at
     configure time. Re-run `make_build.sh` with the export.
   - Toggle it on. Persists in `dev_flags.ini`.

3. Trigger an upload without needing a real desync:
   - **Easiest**: hand-write a manifest into the queue. Launcher will
     pick it up on the next tick.

     ```bash
     # In wsl, against the deployed launcher's game dir.
     # Adjust path to wherever you installed wanwan.
     GAME_DIR='/mnt/c/games/2dfm/wanwan'
     mkdir -p "$GAME_DIR/upload_queue"
     mkdir -p "$GAME_DIR/logs"
     echo "manual probe log" > "$GAME_DIR/logs/FM2K_P1_Debug.log"
     cat > "$GAME_DIR/upload_queue/$(date +%s%N | head -c 13)_desync_p0_42.json" <<'EOF'
     {
       "kind": "desync",
       "frame": 0,
       "session_id": "42",
       "match_id": "launcher-probe",
       "player_index": 0,
       "client_version": "0.2.40-probe",
       "game_id": "WonderfulWorld_ver_0946",
       "hook_dll_sha1": "",
       "rng_seed": "0x00000000",
       "peer_ip": "",
       "timestamp": "2026-05-13T00:00:00Z",
       "files": ["GAMEDIR/logs/FM2K_P1_Debug.log"]
     }
     EOF
     # Substitute GAMEDIR placeholder (the launcher reads absolute
     # paths from the "files" array).
     sed -i "s|GAMEDIR|$(echo $GAME_DIR | sed 's|/mnt/c|C:|; s|/|\\\\|g')|g" \
         "$GAME_DIR/upload_queue/"*.json
     ```

   - With the launcher still running and the checkbox on, watch
     `$GAME_DIR/upload_queue/` — within ~1 second the manifest should
     move to `$GAME_DIR/upload_queue/done/`.
   - If it moves to `quarantine/`: launcher rejected the manifest.
     Reasons appear in the launcher's stderr (or its log file) with
     `UploadQueue:` prefix. Likely cause: malformed JSON or referenced
     files don't exist.
   - If it stays in place: either network failure (open
     `https://hub.2dfm.org/logs/healthz` in a browser to verify) or the
     checkbox isn't actually firing (verify
     `dev_flags.ini` has `auto_upload_logs=1`).

4. Pull it back:
   ```bash
   ./tools/fm2k_logs.py recent --limit 3
   ./tools/fm2k_logs.py show 42 --tail 50
   ```
   You should see the synthetic log content come back.

5. Clean up the test upload from server.

### Real desync test

Once the synthetic path works, generate a real desync:

- Two clients on the same machine using `--p1`/`--p2` flags AND
  intentionally mismatched binaries (e.g. one running v0.2.39, one
  running v0.2.40 — the determinism check probably trips inside the
  first second).
- OR use stress mode: `FM2K_STRESS_MODE=1` on a single instance, which
  runs a determinism check against itself and immediately catches any
  fixed-frame divergence.

The hook should now: (a) terminate the game on desync #1 instead of
accumulating thousands, (b) publish the `FM2K_MATCH_OUTCOME_DESYNC`
shared-mem outcome, (c) drop a manifest in `upload_queue/`. The
launcher should: (a) display a toast about the desync, (b) upload the
manifest within a second.

## 3. Smoke test for remote spectate

The existing runbook lives at `docs/dev/spectator_smoke_test.md`
(8.4 KB, 3 scenarios, written 2026-05-08). Re-run it before iterating
on spectator. Summary of what to expect from each scenario:

| Scenario | Pass criterion | Failure mode → diagnose how |
|---|---|---|
| **Hub-driven Live spectate** | Click an active pair in the launcher's Hub panel → spectator window opens → walks through title/CSS → lands in the host's match, frame-aligned within a second | "stuck on Connecting" → NAT/STUN path. Check spec_hook's STUN probe log line + the hub's `spectate_grant` arrival on the launcher side. |
| **Multi-match validation** | Host + client play 3 sequential matches. Spectator joins on match 2 (FULL_SESSION mode). FF's through match 1, watches match 2 live, watches match 3 live. | Catchup loop locked or `pb_queue` growing unboundedly → live_target / catchup_max_burst_ms issue. Stats: enable `FM2K_DEV_MODE=1` for verbose spectator state lines. |
| **FULL_SESSION regression** | Spectator launched with `--spectate-mode full` on a long session. No mid-set jumps. Parity check at end matches host's final fingerprint. | Fingerprint mismatch at end → input replay drift (bf~8000 sim-determinism leak); pin via `tools/fm2k_logs.py` desync upload if it's automatic, or run `parity_diff.py` against the host's parity log. |

**The auto-upload pipeline we just shipped means scenario 3's "did
the spectator desync at frame N" question now answers itself** — the
spectator hook will publish its desync diff straight to the hub on
divergence, and `tools/fm2k_logs.py show` pulls it down for analysis.

### What to do if remote spectate is *completely* broken

(Most likely failure mode given the v0.2.34→v0.2.39 NAT fix cascade.)

1. Confirm one-machine spectate works first (open a host+client+spec
   on the same WSL/Windows session). If that fails, the bug isn't
   network-layer.
2. If one-machine works but two-machine doesn't: NAT/STUN path. Check:
   - `FM2K_HUB_TCP_STUN_ADDR` is set by the launcher at hub-connect
     time (set in v0.2.36).
   - Spec hook's STUN probe is firing (v0.2.39 was about it silently
     failing).
   - The hub's `spectate_grant` message contains the host's
     STUN-discovered UDP addr (not the WS-source TCP addr — those
     differ on symmetric NATs).
3. Worst case: ship v0.2.40 with the auto-upload pipeline, then USE
   the pipeline to gather logs from one of the users who reported
   spectator broken. Their logs will pinpoint the NAT failure mode
   without us reproducing it locally.

## 4. The spectator improvement plan (the three-iteration sequence)

### Iteration A — verify and ship v0.2.40 as-is

Goal: get the auto-upload pipeline into users' hands so we get real
failure reports.

What ships:
- Boot-to-battle dev checkbox + char/stage/meter overrides
- Crash/desync auto-upload to `hub.2dfm.org/logs`
- Desync immediate-terminate (default on)
- KOF retention HP-init midhook
- Per-game patches v2 (UI in Patches panel)

What doesn't change:
- Spectator code path. FULL_SESSION default stays. CURRENT_MATCH
  still off by default.

Release procedure: `scripts/cut_release.sh "..."` after the §2
verification passes.

### Iteration B — wire `/F` into the spectator launch path

Goal: eliminate the visible "spectator walks title + CSS, then jumps"
experience when joining mid-battle.

Where: `FM2K_RollbackClient.cpp:LaunchRemoteSpectator` (and the
hub-driven `on_spectate_match` path that calls it).

Concrete change:
- When `mode == SpecJoinMode::CURRENT_MATCH`, set
  `FM2K_BOOT_TO_BATTLE=1` for the spectator child's environment.
- Spectator's hook side already primes `g_iniFile_nameOverride` from
  the exe basename — works automatically.
- The deferred-apply gate already waits for `game_mode >= 3000` before
  loading the snapshot; with `/F` the engine reaches that state in
  the first frame, so the snapshot applies immediately with zero
  visible CSS replay.

Risk to verify:
- The kgt TestPlay defaults pick char 0 vs char 0 as the engine's
  placeholder matchup. The snapshot then overwrites the 8×57407-byte
  character-data pool. The `.player` files corresponding to the host's
  *actual* matchup must exist on disk on the spectator's machine
  (same game install assumption — usually fine, worth a sanity check
  in the snapshot-apply log).

Re-flip `CURRENT_MATCH` to default once this is verified.

Effort estimate: half-day. Self-contained.

### Iteration C — snapshot metadata extension for mid-CSS join

Goal: spectator who joins while the host is mid-CSS doesn't have to
replay CSS inputs from frame 0; instead jumps to the host's current
CSS state and naturally walks the rest of CSS to battle entry.

This is what was scoped as "Future task #18 phase B" in the
2026-05-08 commit body.

Sketch:
1. Extend `SnapshotMetadata` with:
   ```cpp
   struct CssSnapshot {
       uint32_t css_frame;        // host's CSS sub-frame count
       uint32_t game_mode_flag;   // 0/1/2
       int32_t  p1_selected_char_idx;
       int32_t  p2_selected_char_idx;
       int32_t  p1_cursor_pos[2]; // grid x,y
       int32_t  p2_cursor_pos[2];
       uint32_t p1_action_state;  // 0 unselected, 1 confirmed
       uint32_t p2_action_state;
       uint32_t p1_round_count;   // for team mode
       uint32_t p2_round_count;
       int32_t  team_chars[2][4]; // team-mode slot picks
   };
   ```

2. Host stashes a CSS snapshot at title→CSS transition (in addition
   to the existing battle-start snapshot) and refreshes it every
   ~30 CSS frames so a late joiner gets a recent state, not frame 0.

3. Spectator that receives a CSS-mode snapshot:
   - Boots without `/F` (need real CSS state to apply it onto).
   - Once `game_mode == 2000` (CSS), applies the CSS-state fields:
     writes `g_p*_selected_char_idx`, `g_p*_cursor_pos`, etc.,
     immediately followed by the engine's natural step.
   - Arms `CssAutoConfirm` to drive the cursor + confirms from the
     forwarded inputs, but starting from the snapshot frame rather
     than frame 0.

Risk to consider:
- CSS state machine has a Setup → Active → Confirmed → Transition
  path (states 0..4 at `obj+338`). Loading a snapshot mid-state has
  to land cleanly — the engine doesn't like being told "you're at
  state 3 with cursor at (5,2)" if it just initialized at state 0
  with cursor at (0,0). May need to drive the state machine via
  synthesized inputs rather than direct memory writes.

Effort estimate: 1-2 days. Bigger than B because we're synthesizing
mid-state CSS, but the CssAutoConfirm machinery already exists to
support driving CSS via forwarded inputs.

## 6. Open questions / followups

- **Do we need symbolicated stack traces** for crash uploads, or is
  `module+RVA` enough? Currently we capture the latter via
  `RtlCaptureStackBackTrace`. To symbolicate we'd need to stash
  `FM2KHook.dll.debug` (unstripped) somewhere by SHA1 and have the
  pull-tool resolve offsets on demand.
- **Should the launcher prompt for opt-in on first launch** of a build
  with `FM2K_LOG_UPLOAD_SECRET` baked? Currently it's a quiet dev
  checkbox; release builds without the secret have it greyed out, so
  release users never see it. Decide whether release builds should
  bake the secret + show the prompt, or release builds should *not*
  bake the secret (release users never auto-upload).
- **Per-IP rate limiting on the receiver.** Server-side caps body
  size and validates schema, but doesn't yet limit "user X uploaded 50
  desyncs today." Easy to add to `app.py` via a sliding window in
  SQLite; not urgent.
- **Retention longer than 30 days for crashes.** Desyncs decay in
  value fast (state corruption that won't repro on a different
  determinism build), but a crash backtrace might be relevant months
  later. Consider separating retention windows per kind.
